# 业务数据数据库定义（PostgreSQL）

## 1. 设计目标

- 支撑 AI 摄影点评核心流程：上传 -> 确认照片 -> 发起点评任务 -> 产出点评结果 -> 查询历史
- 保证数据一致性与可追踪：状态机、幂等键、审计时间戳、失败原因
- 支撑风控与运营：限流计数、配额消耗、模型成本统计

## 2. 命名与约定

- 数据库：PostgreSQL 14+
- schema：`public`
- 主键：`bigserial`（内部自增）+ 外部可暴露 `public_id`（文本）
- 时间：`timestamptz`，统一 UTC
- JSON 扩展字段：`jsonb`
- 软删除：默认不启用，使用状态字段表达业务生命周期

## 3. 实体关系概览

- `users` 1:N `photos`
- `photos` 1:N `review_tasks`
- `photos` 1:N `reviews`
- `review_tasks` 1:1(可空) `reviews`
- `users` 1:N `usage_ledger`
- `users` 1:N `idempotency_keys`
- `users` 1:N `rate_limit_counters`

## 4. 表定义说明

## 4.1 users（用户）

用途：用户账号、套餐、配额与账号状态。

关键字段：
- `public_id`：外部用户 ID（如 `usr_xxx`）
- `plan`：`free/pro/enterprise`
- `daily_quota_total`：日配额
- `daily_quota_used`：日已用
- `status`：`active/suspended/deleted`

约束与索引：
- `email`、`username` 唯一
- `status + plan` 组合查询索引

## 4.2 photos（照片）

用途：记录对象存储 key、EXIF、客户端元信息与安全扫描结果。

关键字段：
- `public_id`：`pho_xxx`
- `owner_user_id`：归属用户
- `upload_id`：上传会话号（来自 presign）
- `bucket`、`object_key`：对象定位
- `status`：`UPLOADING/READY/REJECTED`
- `exif_data`：标准化 EXIF（jsonb）
- `client_meta`：宽高、设备侧信息（jsonb）
- `nsfw_label`、`nsfw_score`：内容安全结果

约束与索引：
- `(bucket, object_key)` 唯一
- `owner_user_id + created_at` 索引
- `status` 索引

## 4.3 review_tasks（点评任务）

用途：异步点评任务状态机、重试和错误追踪。

关键字段：
- `public_id`：`tsk_xxx`
- `photo_id`、`owner_user_id`
- `mode`：`flash/pro`
- `status`：`PENDING/RUNNING/SUCCEEDED/FAILED/EXPIRED`
- `idempotency_key`：客户端幂等键
- `attempt_count`、`max_attempts`
- `error_code`、`error_message`
- `started_at`、`finished_at`、`expire_at`

约束与索引：
- 同一用户同一 `idempotency_key` 唯一（可空）
- `(status, created_at)` 索引（拉取待执行任务）
- `photo_id + created_at` 索引

## 4.4 reviews（点评结果）

用途：存储可展示点评结果（结构化 JSON）及成本数据。

关键字段：
- `public_id`：`rev_xxx`
- `task_id`：来源任务（可空，兼容同步模式）
- `photo_id`、`owner_user_id`
- `mode`：`flash/pro`
- `status`：`SUCCEEDED/FAILED`
- `schema_version`：结果结构版本（如 `1.0`）
- `result_json`：点评结果
- `input_tokens`、`output_tokens`、`cost_usd`
- `latency_ms`

约束与索引：
- `task_id` 唯一（若存在）
- `photo_id + created_at` 索引
- `owner_user_id + created_at` 索引

## 4.5 idempotency_keys（幂等键）

用途：创建类接口防重（如 `POST /reviews`）。

关键字段：
- `user_id`、`endpoint`、`idempotency_key` 唯一
- `request_hash`：请求体哈希
- `response_json`、`http_status`：首次响应缓存
- `expire_at`：过期时间

## 4.6 usage_ledger（用量流水）

用途：统计配额消耗和对账（成功/失败都可记账）。

关键字段：
- `user_id`、`review_id`、`task_id`
- `usage_type`：`review_request/token/image_storage/...`
- `amount`、`unit`、`bill_date`

索引：
- `user_id + bill_date`
- `usage_type + bill_date`

## 4.7 rate_limit_counters（限流计数）

用途：按维度存储限流窗口计数（若使用 Redis，此表可作为持久审计备份）。

关键字段：
- `scope`：`ip/user/endpoint/plan`
- `scope_key`：如 IP、user_id、endpoint 名
- `window_start`、`window_seconds`、`hit_count`

唯一约束：
- `(scope, scope_key, window_start, window_seconds)`

## 5. 状态机规则（数据库层）

- `photos.status`：`UPLOADING -> READY | REJECTED`
- `review_tasks.status`：`PENDING -> RUNNING -> SUCCEEDED | FAILED | EXPIRED`
- `reviews.status`：仅 `SUCCEEDED | FAILED`

建议：状态跃迁由应用层控制，数据库通过枚举和检查约束兜底。

## 6. 数据保留与归档建议

- `review_tasks`：保留 90~180 天，失败任务可长期留存用于排障。
- `usage_ledger`：至少保留 1 年用于对账。
- `rate_limit_counters`：保留 30~90 天。
